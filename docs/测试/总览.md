# 测试总览

本文档汇总 `godot-dotnet-mcp` 当前测试模块的整体结构、覆盖范围、已知边界和后续演进方向。

截至 `2026-03-27`，测试模块已经不再只是零散脚本，而是形成了三层结构：

1. `.NET Host Contract Tests`
2. `Godot Headless Harness`
3. `Central Server Smoke`

这三层对应不同目标：

| 层级 | 位置 | 目标 |
|---|---|---|
| Host contracts | `tests/host_contracts/` | 验证中心服务的工具目录、会话协调、生命周期错误模型与 `centralHostSession` 载荷 |
| Plugin headless harness | `tests/godot_plugin_harness/`、`tests/godot_plugin_harness_fixture/` | 在不依赖第三方测试框架的前提下验证插件运行时、工具装载和路由行为 |
| Central smoke | `central_server/SmokeSystemSession*.cs`、`central_server/smoke/`、`scripts/smoke_central_server_system_session.ps1` | 保留真实链路回归，验证 attach、reuse、lifecycle 与 auto-launch 主线 |

## 当前架构判断

### 已经合理的部分

- 测试已经从“单个 smoke 脚本”演进为三层分工，而不是把所有验证都堆在一个入口里。
- Host 与插件两侧都已经出现独立的支撑层，不再完全依赖单文件入口。
- 插件测试已经开始使用 fake collaborator 验证 `impl_runtime`、`impl_index` 等模块，说明不是所有测试都依赖真实运行时。
- Config 页运行时链已经补入 `serializer / inspection / file transaction / launcher adapter` 四类轻量契约，不再只靠 UI 交互路径兜底。
- smoke 主文件已经明显瘦身，方向正确。

### 当前仍需收口的部分

#### 1. 插件 contract tests 已脱离私有方法名，但 seam 仍可继续收口

当前 `http_server_contract_test.gd` 与 `runtime_bridge_contract_test.gd` 已改为走公共入口，不再直接依赖下划线方法。  
这一轮已经显著降低了“内部方法改名即测试碎裂”的风险，而且 runtime fallback / reply 行为也已经从 bridge 节点里抽出第一轮独立 helper；但 `mcp_http_server.gd` 与 `mcp_runtime_bridge.gd` 仍然持有一部分测试 seam，本质上还可以继续向更独立的 helper 收口。

#### 2. Host contracts 已与 smoke support 解耦，但仍是轻量集成层

当前 Host contract suite 已不再直接复用 `Smoke*Support`。  
不过它仍会组装较完整的 Host 编排链，因此定位上依然更接近“面向对外契约的轻量集成测试”。

#### 3. smoke 场景仍偏厚

虽然 `SmokeSystemSessionRunner.cs` 已经瘦身，但 `SmokeSystemSessionReuseScenario.cs` 和 `SmokeSystemSessionAutoLaunchScenario.cs` 仍然承载了较长的顺序化编排逻辑。

#### 4. CI 中环境依赖测试仍未完全硬门禁

当前 `validate-refactor.yml` 会运行 plugin harness，但允许在缺少 Godot 可执行文件时跳过。  
这意味着插件测试层已经接入 CI，但还没有完全升级为“不可跳过的质量门”。

## 设计原则

当前测试模块的演进原则如下：

- 测试能力是一级架构模块，不再附属于 smoke 脚本或人工真机验证。
- Host contracts、plugin harness、smoke 三层分别承担不同职责，不互相吞并。
- 尽量通过稳定 seam 测试对外行为，而不是依赖私有实现细节。
- 大文件先拆再改，测试文件也遵循相同规则。
- 环境依赖用例必须单独声明门禁策略，不能和快测层混成一类。

## 下一步方向

### 第一优先级

- 继续把 `mcp_http_server.gd` 的稳定测试 seam 从 server 节点内部往外收口：
  - `McpRequestRouter`
  - `EditorLifecycleEndpoint`
  - `ToolListResponseBuilder`
- 继续把 `mcp_runtime_bridge.gd` 的稳定测试 seam 从 bridge 节点内部往外收口：
  - `RuntimeCommandAdapter`
  - `RuntimeFallbackStore`
  - `RuntimeReplyWriter`
  - `RuntimeCaptureExecutor / RuntimeInputExecutor / RuntimeStepExecutor`
- `runtime_control_service.gd` 的第二轮 seam 外提已落地，当前协作者为：
  - `SessionSelector`
  - `EditorErrorMapper`
  - `RuntimeReplyResolver`
  - `RequestCoordinator`
- 这一层后续优先补的是 `runtime_session_lost`、bridge unavailable 和 request cleanup 的负例，而不是再引入新的 reply waiter 抽象。

### 第二优先级

- 继续把 Host contract tests 收口为更清晰的 Host test host / payload assertion / attach fixture 分层。

### 第三优先级

- 把 smoke 场景继续拆为：
  - `fixture setup`
  - `lifecycle assertions`
  - `runtime assertions`
  - `cleanup`

### 第四优先级

- 将测试矩阵继续分层为：
  - `Fast Required`
  - `Integration Required`
  - `Environment Required`

## 结论

测试模块已经从“起步阶段”进入“正式治理阶段”。  
当前最重要的任务，不是继续无序增加 case，而是先把三层测试之间的边界、夹具清理和测试 seam 做稳。

只有在这些基础稳定后，后续测试覆盖的扩展才不会把测试模块重新推回高耦合和超大文件状态。
