# 核心实现

本文档汇总 `localization/`、`plugin/config/`、`plugin/runtime/`、`custom_tools/` 和共享辅助脚本的职责边界，作为“除工具域本体外”的核心实现说明。

此外，v1.0 重构期间的外部 Host 实现位于 `central_server/`，负责 `workspace_*`、`dotnet_*` 与 `system_*` 的统一公开入口。

---

## 目录分层

```text
localization/
plugin/
  ├─ config/
  └─ runtime/
tools/
  ├─ core/
  ├─ *_tools.gd
  └─ <category>/executor.gd
custom_tools/
central_server/
```

## `localization/`

### 组成

- `localization_service.gd`
- `locale_en.gd`
- `locale_zh_cn.gd`
- `locale_zh_tw.gd`
- `locale_ja.gd`
- `locale_de.gd`
- `locale_es.gd`
- `locale_fr.gd`
- `locale_pt.gd`
- `locale_ru.gd`

### 职责

- 加载所有语言字典
- 切换当前语言
- 提供 `get_text()` 与语言显示名辅助
- 为 Dock UI 和工具 profile 文案供值

### 约束

- 语言文件键集合应与 `locale_en.gd` 对齐
- 场景文件中的固定文案不应作为最终显示来源
- 插件内部根控件普遍关闭 `auto_translate_mode`

### 状态与生命周期

- `plugin_runtime_state.gd`
- `plugin_runtime_state_service.gd`
- `plugin_tool_bridge_service.gd`
- `tool_permission_policy.gd`
- `tool_profile_catalog.gd`
- `plugin_settings_schema.gd`
- `server_runtime_controller.gd`
- `plugin_reload_coordinator.gd`

### 诊断与调试

- `plugin_self_diagnostic_store.gd`
- `mcp_runtime_debug_store.gd`
- `mcp_editor_debugger_bridge.gd`：注册到 `EditorDebuggerPlugin`，接收主项目调试会话事件，转发至 `MCPDebugBuffer` 和 `MCPRuntimeDebugStore`
- `mcp_runtime_bridge.gd`：以 Autoload 形式安装到主项目，主项目运行时可通过 `emit_event` / `emit_log` / `emit_info` / `emit_warning` / `emit_error` 发送结构化事件；双通道（`godot_mcp/runtime_event` + `godot_mcp/runtime_log`），最多缓存 300 条，EngineDebugger 不活跃时 fallback 写入 `user://godot_mcp_runtime_bridge_events.json`

### 服务与目录

- `mcp_http_server.gd`
- `mcp_tool_loader_supervisor.gd`
- `mcp_tool_rpc_router.gd`
- `tool_catalog_service.gd`
- `user_tool_service.gd`

这些对象共同支撑：

- 设置与 profile 投影
- 内嵌 server 生命周期
- tool loader 生命周期、自恢复与状态监督
- `tools/list` / `tools/call` 的工具路由与结果归一化
- 工具目录与权限裁剪
- User 工具发现与管理
- 自诊断链路

## `tools/` 共享层

除了各 category executor 外，`tools/` 目录还包含一批共享脚本：

### 基础与工具兼容层

- `base_tools.gd`
- `plugin_shared.gd`
- `scene_tools.gd`
- `node_tools.gd`
- `script_tools.gd`
- 其它 `*_tools.gd`

### 通用辅助

- `mcp_debug_buffer.gd`
- `mcp_file_utils.gd`
- `mcp_json_utils.gd`
- `mcp_node_utils.gd`
- `mcp_script_parser.gd`
- `mcp_type_utils.gd`

### 核心

- `tool_manifest.gd`
- `tool_manifest_data.gd`
- `core/tool_loader.gd`
- `core/tool_diagnostic_service.gd`
- `core/tool_exposure_service.gd`
- `core/tool_metrics_service.gd`
- `core/tool_runtime_service.gd`
- `core/tool_reload_service.gd`

这些脚本的关系是：

- `tool_manifest_data.gd` 定义“静态目录事实源是什么”
- `tool_manifest.gd` 负责“如何稳定读取这些事实源”
- `tool_loader.gd` 直接从 `tool_manifest.gd` 读取运行时元数据，并负责“怎么装、怎么调”
- `tool_diagnostic_service.gd` 负责“怎么聚合 load error、duplicate category 与 reload incident”
- `tool_exposure_service.gd` 负责“怎么投影成当前可见工具目录与状态”
- `tool_metrics_service.gd` 负责“怎么累计启动/reload/调用统计并输出性能视图”
- `tool_runtime_service.gd` 负责“怎么加载 executor runtime、缓存 definitions、卸载 domain”
- `tool_reload_service.gd` 负责“怎么完成 domain reload、失败回滚与 reload 后刷新”
- `base_tools.gd` 和其它 helper 负责“怎么少重复代码”

## `custom_tools/`

这个目录承载用户通过 `plugin_evolution` 工具或脚手架维护的 User 工具脚本。当前约束是：

- 不写入内置工具域目录
- 通过 `plugin_evolution` 两步授权流程创建或删除
- 由 `system/executor.gd._load_custom_tools()` 扫描发现
- 删除时可写入 `.backup/` 以支持恢复

## 核心设计取向

当前插件实现强调四个原则：

- 薄入口：`plugin.gd` 只做装配和桥接
- 场景化 UI：Dock 页签单独成场景和控制器
- 域拆分工具：按 category executor 划分而不是单个大工具
- 结构化观测：错误、操作、会话与日志尽量结构化记录
- 权限单一入口：`tool_access_feature.gd` 作为插件侧唯一 permission provider

Host 侧当前强调三个原则：

- 单一入口：外部只连接 `Central Server`
- 会话协调：先解析项目上下文，再决定复用/拉起 Godot
- 结构化错误：项目缺失、可执行路径缺失、attach 超时分别返回独立错误码
- 状态可读：`system_*` 成功返回时补充 `centralHostSession`，直接暴露 active project / attached session 摘要

围绕 Godot 可执行路径，当前还补了一条长期产品约束：

- 中心服务只做常见目录发现，不做更广范围搜索
- 如果没有路径，Host 应返回结构化 guidance，驱动客户端或 Agent 显式询问用户提供 Godot 编辑器路径
- 机器相关 Godot 路径只允许进入用户态配置或本次验收命令参数，不允许进入发布产物或仓库默认配置

当前 Host auto-launch 链路还补了一层运行时端点传递：

- `EditorProcessService` 在拉起真实 Godot 时，会注入：
  - `GODOT_DOTNET_MCP_CENTRAL_SERVER_HOST`
  - `GODOT_DOTNET_MCP_CENTRAL_SERVER_PORT`
- 插件侧 `central_server_attach_service.gd` 会优先读取这两个环境变量，从而保证非默认 attach 端口下的 auto-launch 与烟测都能稳定回连 Host。
- `EditorProcessService` 还会为每次 auto-launch 临时分配一个可用的编辑器 MCP HTTP 端口，并注入：
  - `GODOT_DOTNET_MCP_SERVER_HOST`
  - `GODOT_DOTNET_MCP_SERVER_PORT`
- `server_runtime_controller.gd` 与 `central_server_attach_service.gd` 会优先读取这两个覆盖值，避免本机默认 `3000` 端口被占用时导致 attach 超时。
- 为避免发布验收脚本被编辑器控制台输出拖住，`EditorProcessService` 现在会把 Godot 的标准输出和标准错误重定向到后台读取，而不是继承外层命令管道。

围绕这条链路，当前的长期护栏也已经明确：

- `scripts/smoke_central_server_system_session.ps1 -SkipBuild`
  - 作为长期 CI 护栏，验证 reuse session 主路径
  - 默认使用仓库内隔离 smoke 状态目录，不污染真实用户配置
- `scripts/smoke_central_server_system_session.ps1 -AutoLaunch -ProjectRoot <path> [-GodotExecutablePath <path>]`
  - 作为长期集成烟测，验证真实 auto-launch 主路径
- `-RequireAutoLaunch`
  - 作为长期发布验收开关，要求 auto-launch 不能以 `skipped=true` 通过
- `scripts/validate_central_server_release.ps1 -ProjectRoot <path> -GodotExecutablePath <path>`
  - 作为长期发布机标准化验收入口，统一输出一份 health / reuse smoke / auto-launch smoke 汇总 JSON
