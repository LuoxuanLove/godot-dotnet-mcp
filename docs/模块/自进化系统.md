# 自进化系统

## 概览

自进化系统用于管理插件中的 `User` 工具。

这套系统允许客户端在授权前提下创建、加载、审计和删除用户自定义工具，同时保持以下边界：

- 只管理 `User` 分类工具
- 不写入内置分类
- 创建和删除都要求显式授权
- 每次写操作都会留下审计记录

## 适用范围

自进化系统适合以下场景：

- 需要为当前项目补一个很小的专用工具
- 想把重复使用的工作流固化成可调用工具
- 需要在不改动内置工具的前提下扩展 MCP 能力

不适合以下场景：

- 替代内置工具分类
- 大规模重构插件本体
- 绕过权限边界直接写入内置目录

## 当前组成

自进化系统当前包含四部分：

| 组成 | 作用 |
|---|---|
| `custom_tools/` | User 工具脚本目录 |
| `User` 分类 | Dock 中展示和管理用户工具的入口 |
| `plugin_evolution_*` | 创建、删除、审计和查看 User 工具的工具族 |
| 审计记录 | 记录预演、授权创建、授权删除及失败结果 |

## 相关工具

### `plugin_evolution`

自进化系统相关的 MCP 工具包括：

- `plugin_evolution_list_user_tools`
- `plugin_evolution_scaffold_user_tool`
- `plugin_evolution_delete_user_tool`
- `plugin_evolution_user_tool_audit`
- `plugin_evolution_usage_guide`

### `plugin_runtime`

自进化工具生效后，通常还会配合以下工具查看状态：

- `plugin_runtime_state`
- `plugin_runtime_reload`
- `plugin_runtime_toggle`

### `plugin_developer`

如果需要在 Dock 中显示或隐藏 `User` 分类，可使用：

- `plugin_developer_user_visibility`

## 授权规则

自进化系统的写操作默认不是直接落地，而是先走预演。

当前规则如下：

- 创建 User 工具时，未授权会返回预演结果，不会写文件
- 删除 User 工具时，未授权会返回预演结果，不会删文件
- 只有在 `authorized=true` 时，创建或删除才会真正执行
- 写操作成功或失败都会写入审计记录

## 标准工作流

### 1. 查看当前已有的 User 工具

推荐先调用：

- `plugin_evolution_list_user_tools`

它会返回当前已注册的 User 工具、脚本路径和对应工具名，适合在新增前先确认是否已有同类工具。

### 2. 预演创建

创建前先调用：

- `plugin_evolution_scaffold_user_tool`

建议先传入：

- `tool_name`
- 可选的 `display_name`
- 可选的 `description`
- `authorized=false`

此时返回结果会给出预期脚本路径和工具信息，但不会真正写入文件。

### 3. 授权创建

确认预演结果无误后，再次调用：

- `plugin_evolution_scaffold_user_tool`

并传入：

- `authorized=true`

创建成功后，插件会自动刷新工具目录，使新的 User 工具进入最新状态。

### 4. 查看审计记录

每次创建或删除之后，建议调用：

- `plugin_evolution_user_tool_audit`

它适合用于确认：

- 本次操作是否经过授权
- 本次操作是否成功
- 返回的预演信息和实际执行信息是否一致

### 5. 删除工具

删除时调用：

- `plugin_evolution_delete_user_tool`

建议顺序：

1. 先用 `plugin_evolution_list_user_tools` 确认脚本路径。
2. 再用 `authorized=false` 做一次预演。
3. 最后用 `authorized=true` 执行删除。
4. 删除后再看一次审计记录。

## 文件与目录

当前 User 工具默认放在：

```text
res://addons/godot_dotnet_mcp/custom_tools/
```

创建脚手架后，会生成对应的 `.gd` 脚本文件；删除时会同时清理对应的 `.uid` 文件。

## Dock 中的表现

当前 `User` 分类默认受开发者可见性控制。

常见用法：

1. 用 `plugin_developer_user_visibility` 控制是否在 Dock 中显示 `User` 分类。
2. 在 Tools 页签中查看当前 User 工具是否已出现。
3. 配合工具预设决定是否默认暴露 User 分类。

## 审计记录

自进化系统会记录最近的创建和删除行为，包括：

- 操作类型
- 是否授权
- 是否成功
- 错误码
- 预演或实际执行的目标信息

审计记录更适合做以下工作：

- 回看最近一次写操作是否真正落地
- 区分“未授权预演”与“已授权执行”
- 排查脚本已存在、目录创建失败或删除失败等问题

## 典型建议

- 创建前总是先做一次预演。
- 不要把内置工具的维护工作混入 `User` 分类。
- 每次授权创建或删除后，都检查一次审计记录。
- 如果新工具需要影响主项目运行结果，可再结合 `debug_runtime_bridge` 回看最近一次运行状态。

## 当前限制

- 自进化系统当前只管理 `User` 分类，不会把工具写入内置分类。
- 它更适合生成项目专用的小工具，不适合作为大规模插件开发入口。
- 如果项目里长期积累较多 User 工具，建议定期清理不再使用的脚本和预设。
