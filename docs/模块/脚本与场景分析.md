# 脚本与场景分析

## 脚本工具

当前脚本工具接口包括：

- `script_read`
- `script_open`
- `script_inspect`
- `script_symbols`
- `script_exports`
- `script_edit_gd`

### 能力边界

- `.gd` 与 `.cs` 都支持读取、打开、结构分析
- `.gd` 支持有限编辑
- `.cs` 当前定位是只读分析，不承担复杂改写

## `script_*` 操作说明

### `script_read`

用途：

- 读取 `.gd` 或 `.cs` 原文
- 快速确认脚本文件路径是否正确
- 为外部客户端补上下文

推荐场景：

- 需要让客户端看到完整脚本文本
- `script_inspect` 的结构化结果不足以解释问题

### `script_open`

用途：

- 在 Godot 编辑器脚本页签中打开脚本
- 跳到指定行
- 查看当前已打开的脚本列表

推荐场景：

- 从 MCP 调用切回手工编辑
- 把分析结果快速定位到编辑器中的实际代码位置

### `script_inspect`

用途：

- 返回语言、类名、基类、方法、导出成员、导出分组

推荐场景：

- 想快速知道脚本暴露了什么
- 需要在不打开脚本编辑器的情况下做静态结构检查

### `script_symbols`

用途：

- 以统一结构返回 class / method / export / enum
- 支持按种类和关键字过滤

推荐场景：

- 构建脚本索引
- 根据导出成员或方法名做快速筛选

### `script_exports`

用途：

- 只返回导出成员和导出分组

推荐场景：

- 准备和场景绑定做交叉比对
- 只关心 Inspector 可见字段，不需要完整符号表

### `script_edit_gd`

用途：

- 仅对 GDScript 提供有限编辑

支持动作：

- 新建脚本
- 整体重写
- 增删函数
- 增加变量、信号、导出字段
- 读取函数列表和变量列表

限制：

- 不承担任意 AST 级改写
- 不处理 C# 编辑
- 更适合模板化小改动，不适合大规模重写

## 场景分析工具

当前场景分析工具包括：

- `scene_bindings`
- `scene_audit`

### 目标问题

它们主要用于发现以下问题：

- `[Export]` 成员未绑定
- 场景绑定值缺失
- 绑定类型与预期不匹配
- 脚本存在但导出成员未正确暴露

## `scene_*` 分析操作说明

### `scene_bindings`

用途：

- 分析当前场景或指定场景中，脚本导出成员的实际绑定状态

推荐流程：

1. 使用 `scene_bindings.current` 或 `scene_bindings.from_path`。
2. 查看每个节点的脚本路径、导出成员、当前绑定值、是否缺失。
3. 如需核对脚本声明，再调用 `script_exports` 或 `script_inspect`。

### `scene_audit`

用途：

- 将 `scene_bindings` 的结果进一步收敛为结构化问题列表

推荐流程：

1. 使用 `scene_audit.current` 或 `scene_audit.from_path`。
2. 关注 `issue_count` 和 `issues`。
3. 针对问题中的节点路径，再调用 `scene_hierarchy`、`node_query` 或脚本工具追查。

## 与 Godot.NET 的关系

这组工具的价值不在于“读脚本文本”，而在于把脚本结构和场景实例绑定关系连接起来。对于 Godot.NET 项目，这正是手工排查最耗时的部分。

## 典型排障流程

### 导出字段为空

1. 用 `script_exports` 看脚本里是否真的声明了 `[Export]` 或 `@export`。
2. 用 `scene_bindings` 看场景实例里该字段是否为空。
3. 用 `scene_audit` 获取结构化问题描述。
4. 如需定位节点，再用 `scene_hierarchy` 或 `node_query.get_info`。

### 脚本换基类后 Inspector 异常

1. 用 `script_inspect` 确认当前 `base_type`。
2. 用 `scene_bindings` 观察已有导出绑定是否还可解析。
3. 如果绑定消失，检查场景是否需要重新保存或重新绑定。

### 想从脚本声明回跳到编辑器

1. 用 `script_symbols` 或 `script_inspect` 找到方法或导出成员。
2. 用 `script_open.open_at_line` 直接跳到对应脚本。

## 实现方法

### 脚本分析

脚本分析主入口位于 `tools/script_tools.gd`。当前实现路线是：

1. 读取文件文本。
2. 根据扩展名判断 GDScript / C#。
3. 调用 `base_tools.gd` 中的脚本元数据解析 helper。
4. 将结果整理为 `language`、`class_name`、`methods`、`exports`、`symbols` 等字段。

这样做的原因是：

- `.gd` 和 `.cs` 需要共用一套对外返回结构
- 外部客户端通常只关心“能否拿到类、方法、导出”，不需要完整编译语义

### 场景绑定分析

场景分析主入口位于 `tools/scene_tools.gd`。当前流程是：

1. 定位当前编辑场景或按路径加载场景。
2. 遍历节点。
3. 读取节点脚本及其导出成员定义。
4. 对每个导出成员提取当前场景实例值。
5. 生成绑定记录和问题列表。

这条链路的关键点在于：它不是只看场景文件文本，也不是只看脚本文本，而是以“脚本声明 + 场景实例值”的组合结果为准。

## 当前风险点

如果以下任一情况出现，应优先视为实现缺陷而不是用户误用：

- 脚本工具能解析导出成员，但场景分析无法读取同一成员
- 场景分析能发现节点，却无法稳定解析同一路径给节点工具
- 返回结构里缺失绑定值类型，导致客户端无法判断是否真实绑定
