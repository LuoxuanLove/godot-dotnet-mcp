# 工具系统

## 概览

`Godot .NET MCP` 的工具系统按能力域拆分，每个工具族负责一类明确工作流，例如场景管理、节点修改、脚本分析、资源处理或插件运维。

当前版本的工具目录更强调三点：

- 分类稳定：优先按能力域找工具，而不是在单个大工具里堆叠过多动作
- 写后可验证：大多数写操作都应配套读回或查询工具
- 面向真实使用：优先覆盖 Godot 编辑器内的常见操作，而不是追求理论上的 API 全覆盖

## 当前分组

Tools 页签当前按以下根分组展示：

| 根分组 | 主要分类 | 典型用途 |
|---|---|---|
| `Core` | `intelligence`、`scene`、`node`、`script`、`resource`、`filesystem`、`project`、`editor`、`debug`、`group`、`signal` | Intelligence 高层工具、场景、节点、脚本、资源、项目与编辑器日常操作 |
| `Plugin` | `plugin_runtime`、`plugin_evolution`、`plugin_developer` | 插件运行状态、User 工具管理、开发者控制项 |
| `Visual` | `animation`、`material`、`shader`、`lighting`、`particle`、`tilemap`、`geometry` | 动画、渲染、视觉资源和关卡可视内容 |
| `Gameplay` | `physics`、`navigation`、`audio` | 物理、寻路、音频和运行时行为 |
| `Interface` | `ui` | UI 结构、主题与布局 |
| `User` | `user` | 通过用户扩展机制加载的自定义工具 |

## 主要工具分类

### `intelligence`

Intelligence 层是 Agent 使用本插件的推荐起点，提供项目状态、场景分析、脚本结构检查和符号索引四类高层能力。

15 个内置工具分为四组：

**项目级**：`intelligence_project_state`、`intelligence_project_advise`、`intelligence_runtime_diagnose`、`intelligence_project_configure`、`intelligence_project_run`、`intelligence_project_stop`

**场景级**：`intelligence_scene_validate`、`intelligence_scene_analyze`、`intelligence_scene_patch`

**脚本级**：`intelligence_bindings_audit`、`intelligence_script_analyze`、`intelligence_script_patch`

**索引级**：`intelligence_project_index_build`、`intelligence_project_symbol_search`、`intelligence_scene_dependency_graph`

与原子工具的关系：Intelligence 工具通过 Atomic Bridge 组合调用下层原子 executor，适合"理解上下文 → 分析问题 → 批量修改"的完整工作流；原子工具适合已知目标的精细操作。

详细说明见 [Intelligence工具层.md](Intelligence工具层.md)。

### `scene`

场景相关工具用于打开、保存、运行、查看层级，以及做脚本绑定审计。

当前常用工具包括：

- `scene_management`
- `scene_run`
- `scene_hierarchy`
- `scene_bindings`
- `scene_audit`

适合场景：

- 打开或切换当前编辑场景
- 读取当前选中节点和层级结构
- 对 C# / GDScript 导出字段做场景绑定检查

### `plugin_evolution`

`plugin_evolution` 负责 User 工具的完整生命周期管理。

全部 7 个工具：

- `plugin_evolution_list_user_tools` — 列出所有已注册 User 工具
- `plugin_evolution_scaffold_user_tool` — 预览或创建 User 工具脚本框架（两步授权）
- `plugin_evolution_delete_user_tool` — 预览或删除 User 工具脚本（两步授权）
- `plugin_evolution_restore_user_tool` — 预览或恢复最近删除的 User 工具（两步授权）
- `plugin_evolution_user_tool_audit` — 读取审计记录
- `plugin_evolution_check_compatibility` — 对比现有 User 工具与当前脚手架版本
- `plugin_evolution_usage_guide` — 返回推荐的授权工作流说明

详细说明见 [用户扩展.md](用户扩展.md)。

- `plugin_developer_apply_profile` — 应用指定工具预设
- `plugin_developer_save_profile` — 将当前工具选择保存为自定义预设
- `plugin_developer_rename_profile` — 重命名自定义预设
- `plugin_developer_delete_profile` — 删除自定义预设
- `plugin_developer_export_config` — 导出当前 profile 与禁用工具选择到 JSON
- `plugin_developer_import_config` — 从 JSON 导入 profile 与禁用工具选择并应用
- `plugin_developer_usage_guide` — 返回推荐的使用流程与调试说明

## 兼容入口

当前仍保留 `plugin` 分类下的兼容聚合入口：

- `plugin_runtime`
- `plugin_evolution`
- `plugin_developer`

新接入、文档和脚本示例均推荐优先使用拆分后的细粒度工具名，而不是继续扩展兼容聚合入口。

## 典型使用方式

### 先查再改再验证

推荐顺序：

1. 先用查询工具确认目标路径、状态或文件存在性。
2. 再执行写操作。
3. 最后立即用对应读取工具验证结果。

示例：

1. 用 `node_query` 或 `scene_hierarchy` 找到目标节点。
2. 用 `node_property`、`filesystem_json` 或 `animation_track` 执行修改。
3. 用 `node_property`、`filesystem_json` 或 `animation_track` 读回确认。

### 插件和 User 工具操作

如果涉及插件自身状态或 User 工具：

1. 先用 `plugin_runtime_state` 查看当前状态。
2. 再用 `plugin_runtime_toggle` 或 `plugin_runtime_reload` 做调整。
3. 如需处理 User 工具，再使用 `plugin_evolution_*`。
4. 变更完成后可再次读取状态或审计记录确认结果。

### 运行时问题回读

如果问题发生在主项目运行阶段：

1. 正常从编辑器运行主项目。
2. 停止后使用 `intelligence_runtime_diagnose`。
3. 读取运行时错误、编译问题与性能快照。

## 输入与返回约定

### 工具名格式

当前工具系统统一使用：

- 工具名：`{category}_{tool}`
- 工具参数：按各工具自己的字段定义传入

例如：

- `scene_management`
- `plugin_runtime_reload`
- `plugin_evolution_scaffold_user_tool`

### 常见路径写法

- 资源路径使用 `res://`
- 节点路径优先使用当前场景根节点下的相对路径
- `/root/...` 风格路径在当前版本也可用

### 常见返回结构

工具返回结果默认包含以下字段：

- `success`
- `message`
- `data`

失败时会返回：

- `success: false`
- `error`

## 使用建议

- 新增工作流优先寻找现有分类，不要先假设需要新工具。
- 修改项目内容时，优先选择可以“写后读回”的工具组合。
- 需要插件说明时，可直接使用三个 `usage_guide` 工具。
- 处理 User 工具前，先阅读 [用户扩展.md](用户扩展.md)。
