# 工具系统

## 设计原则

工具系统遵循三条规则：

- 一个能力域只保留一个权威分类
- 工具名保持稳定，域内再用 `action` 分派行为
- 优先服务真实编辑器工作流，而不是理论上的 API 全覆盖

补充约束：

- 写操作尽量提供读回路径
- 共享输入格式必须跨工具一致
- 不同工具对同一节点路径的解析结果必须一致

## 当前核心分类

当前主要分类包括：

- `scene`
- `node`
- `resource`
- `filesystem`
- `project`
- `editor`
- `debug`
- `script`
- `animation`
- `physics`
- `navigation`
- `audio`
- `ui`

每个分类由一个执行器脚本承接，例如：

- `scene_*` -> `tools/scene_tools.gd`
- `node_*` -> `tools/node_tools.gd`
- `script_*` -> `tools/script_tools.gd`
- `editor_*` -> `tools/editor_tools.gd`

## 收敛后的边界

以下旧入口已不再作为独立分类继续扩展：

- `signal_*`
- `group_*`
- `node_physics`

收敛规则如下：

- 信号与分组相关能力应归入 `node`
- 物理能力应统一归入 `physics`
- 同义工具不应长期并存

## 调用风格

推荐调用模式：

1. 先用查询工具确认路径、资源或状态
2. 再执行写操作
3. 最后立即用读取工具验证结果

对写操作而言，“执行成功”不等于“结果可信”，必须有读回闭环。

推荐闭环示例：

1. 用 `scene_hierarchy` 或 `node_query` 找到目标节点。
2. 执行 `node_property.set`、`filesystem_json.write`、`animation_track.add_key` 等写操作。
3. 立即用 `node_property.get`、`filesystem_json.get_value`、`animation_track.list` 等读取工具回查。

## 返回格式要求

工具实现需要保证：

- 失败时返回明确 `error`
- 可恢复错误尽量附带 `hints`
- 成功返回中避免无语义空对象
- 返回字段命名尽量稳定，避免相同语义多种写法

当前约定补充如下：

- 成功返回固定为 `success + data + message`
- 失败返回固定为 `success:false + error`，可选 `data` 与 `hints`
- `plugin/runtime/mcp_http_server.gd` 会在协议出口做最终归一化，避免历史工具返回差异直接暴露给客户端

## 输入契约

### 工具名与 action

当前工具系统默认使用两层路由：

- 第一层：`{category}_{tool_name}`
- 第二层：`action`

例如：

- `scene_management` + `open`
- `editor_status` + `get_main_screen`
- `animation_track` + `add_key`

### 节点路径规则

共享路径解析位于 `base_tools.gd._find_node_by_path()`。当前接受并等价处理：

- 空字符串
- `/`
- 当前编辑场景根节点名
- 以根节点名开头的相对路径
- 编辑器树绝对路径
- 当前场景中的子节点相对路径

这条规则的目的，是避免“树里能看到、工具里找不到”的根节点路径不一致问题。

### 可见性字段

共享节点序列化位于 `base_tools.gd._node_to_dict()`。对 `CanvasItem` / `Node3D`，当前返回：

- `visible`：节点本地可见开关
- `visible_in_tree`：叠加父链后的最终可见状态

非可见性节点不再伪造 `visible: true`。

### JSON / Variant 归一化

共享值归一化位于 `base_tools.gd._normalize_input_value()`。当前规则：

- 如果输入是 JSON 字符串，先解析
- 如果能拿到参考值，则按参考值类型转换
- 校验以“转换后的值”是否匹配目标类型为准

已复用到：

- `filesystem_json.write`
- `filesystem_json.set_value`
- `animation_track.add_key`
- `animation_tween.property`

## 典型操作说明

### 修改节点属性

1. 先用 `node_query.get_info` 或 `node_property.list` 确认目标路径和属性名。
2. 用 `node_property.set` 执行修改。
3. 用 `node_property.get` 读回。
4. 如果涉及显隐，再用 `node_query.get_info` 或 `find_children` 对照 `visible` / `visible_in_tree`。

### 修改 JSON 配置

1. 用 `filesystem_json.read` 读取现有文件。
2. 用 `filesystem_json.write` 或 `set_value` 写入对象、数组或 JSON 字符串。
3. 立即用 `get_value` 验证点路径结果。

### 修改动画数据

1. 先用 `animation_player.list` 或 `animation_track.list` 确认动画名和轨道。
2. 用 `animation_track.add_key` 写入值。
3. 如果是 BlendSpace，写后立即用 `animation_blend_space.get_points` 验证新增点索引和位置。

## 实现方法

### 共享基础层

`base_tools.gd` 负责：

- 成功/失败返回构造
- `res://` 路径标准化
- 当前编辑场景访问
- 节点路径解析
- 节点序列化
- 常见 Variant 类型转换
- JSON 类字符串解析
- 嵌套字典点路径读写

这层存在的原因是把跨分类的一致性强制前置，避免每个工具自己发明一套路径或 JSON 规则。

本轮还新增了一条要求：

- `_success()` 必须始终返回 `data` 与 `message` 字段，即使数据为空也不省略结构

### 为什么 `filesystem_json.write` 必须先做归一化

如果客户端传的是 JSON 字符串而不是 Dictionary，直接再次 `JSON.stringify()` 会把对象写成字符串文本。当前实现先解析，再写盘，目的是保证：

- 写后文件仍然是合法 JSON 结构
- `get_value` 能立刻读取到对象字段
- 客户端可以混用“直接对象”和“JSON 字符串”两种输入方式

### 为什么动画工具要按参考类型转换

Godot 动画轨道的目标属性可能是：

- `Vector2`
- `Vector3`
- `Color`
- 标量数值

如果只按输入字典外形判断，很容易把 `Node3D.position` 误判成二维值。当前实现优先读取目标属性或轨道参考值，再决定如何反序列化和校验。

### 为什么协议层还要再做一次结果归一化

历史工具中仍可能存在：

- 成功但缺失 `data`
- 成功字段之外直接挂业务字段
- 成功响应没有 `message`

当前服务层会在 `tools/call` 出口统一做一次归一化，目的不是替代工具层，而是保证外部 MCP 客户端始终拿到稳定结构。

## 开发约束

- 新增工具前先确认是否已有同义 category。
- 新增写操作前先设计对应读回验证路径。
- 新增共享字段时，先检查是否应下沉到 `base_tools.gd`。
- 变更路径规则、可见性字段或 JSON 归一化逻辑时，必须同步更新测试和迁移文档。
