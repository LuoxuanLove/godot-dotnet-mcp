# 服务与路由

## 服务入口

服务层由 `plugin/runtime/mcp_http_server.gd` 承接，运行在 Godot 编辑器进程内。它负责监听客户端请求，并将协议消息路由到已注册的工具执行器。

当前入口关系如下：

- `plugin.gd` 不直接处理协议，只负责装配和启停
- `plugin/runtime/server_runtime_controller.gd` 负责创建、重建、停止和释放 `MCPHttpServer`
- `plugin/runtime/mcp_http_server.gd` 负责网络监听、请求解析和工具分发

## 路由规则

外部工具名使用：

```text
{category}_{tool_name}
```

例如：

- `scene_management`
- `node_property`
- `script_inspect`
- `physics_physics_body`

服务层根据工具名前缀找到对应执行器，再由执行器内部的 `action` 分派具体行为。

对应实现：

1. `tool_registry.gd` 声明内置 `category -> script path`，并扫描 `custom_tools/*.gd`。
2. `tools/core/tool_loader.gd` 负责定义缓存、运行时实例、懒加载和热重载。
3. 每个执行器必须实现 `get_tools()` 和 `execute()`；支持 runtime 扩展的执行器可额外实现 `configure_runtime()`。
4. 服务启动或重初始化时，工具定义会统一收集为 `{category}_{name}`。

## HTTP 入口

当前服务内置以下入口：

| 方法 | 路径 | 作用 |
|---|---|---|
| `POST` | `/mcp` | MCP 主入口 |
| `GET` | `/health` | 基本健康检查 |
| `GET` | `/api/tools` | 当前工具列表 |
| `OPTIONS` | `*` | CORS 预检 |

额外约束：

- `GET /mcp` 会返回 `405`，并通过 `Allow` 明确只允许 `POST, OPTIONS`
- 监听地址由 `host + port` 决定，当前默认是 `127.0.0.1:3000`
- 连接统计和最近请求信息由服务进程内内存维护，不持久化到磁盘

## 请求处理流程

一次典型 `tools/call` 流程如下：

1. TCP 连接接入。
2. `plugin/runtime/mcp_http_server.gd` 在 `_process()` 中累积 HTTP 数据。
3. 服务解析请求头、`Content-Length` 或 `chunked` 传输体。
4. 如果命中 `/mcp`，则进入 MCP 消息处理流程。
5. 服务从工具名中定位 category。
6. `MCPToolLoader` 返回当前有效执行器实例；必要时按需加载。
7. category 执行器收到参数，进入 `execute(tool_name, args)`。
8. 执行器返回统一字典结构。
9. 服务层再把结果包装为 MCP `content[].text`，并设置 `isError`。

## 返回结构

成功返回：

```json
{
  "success": true,
  "data": {},
  "message": ""
}
```

失败返回：

```json
{
  "success": false,
  "error": "",
  "data": {},
  "hints": []
}
```

MCP 层再将其包装到 `content[].text`，并根据 `success` 设置 `isError`。

## 路由与禁用工具

服务层支持禁用工具列表，来源是 Dock `Tools` 页签维护的 `disabled_tools`。

行为规则如下：

- 服务注册工具时不会删除定义，而是在对外暴露和执行时按名称判断是否可用
- 启动阶段会缓存所有 category 的工具定义，但只常驻启用 category 的运行时实例
- profile 切换后，`plugin.gd` 会调用 `server_runtime_controller.set_disabled_tools()`
- 重启或完全重载插件时，禁用列表会一并重新注入 `MCPServer`

这保证了两件事：

- UI 和服务端对“当前可用工具”有同一份来源
- 不需要为禁用工具单独维护第二套路由表

## 错误语义

当前协议约束：

- 工具调用成功时，MCP 层应返回 `isError: false`
- 工具调用失败时，MCP 层应返回 `isError: true`
- 业务字段错误不应伪装成“成功但空结果”

对客户端而言，以下情况都应视为异常并单独修复：

- `success: true`，但没有可用的 `data`
- 写操作成功后无法被查询工具读回
- 同一节点路径在不同工具里解析结果不一致
- 工具返回空对象 `{}`，但没有可解释的状态含义

## 注册与收敛原则

- 一个能力域只保留一个权威入口
- 不再长期维护重复语义的双轨 API
- 兼容迁移时可保留旧路径文档，但新实现应统一落到收敛后的分类

## 实现方法

### 工具注册

当前实现采用“显式注册表 + 受控自定义扫描”：

- 内置工具通过 `tool_registry.gd` 维持稳定顺序和脚本入口
- 自定义工具只扫描 `custom_tools/*.gd`，避免全项目目录遍历
- 所有 category 的 load state、reload status 和 load errors 都由 `MCPToolLoader` 汇总

### 协议封装

服务层把执行器结果统一包装，是为了避免每个工具脚本重复关心：

- HTTP 状态码
- JSON-RPC / MCP `result` 结构
- `isError` 语义
- `content[].text` 的最终序列化

### 为什么仍然保留 `GET /api/tools`

这是一个调试与排障入口，主要用途是：

- 在不经过 MCP 客户端的情况下查看当前服务是否正常注册工具
- 快速确认 `disabled_tools` 是否生效
- 验证某次重构是否意外丢失 category

## 排障建议

- 如果服务显示已启动但客户端无法连接，优先检查端口是否被占用和 `GET /health` 是否可达。
- 如果某一类工具突然全部消失，先看 `_tool_load_errors` 是否出现脚本加载失败。
- 如果热重载后行为未更新，先调用 `plugin_runtime` 查看 `reload_status` 与 domain `load_state`。
- 如果 `tools/call` 看起来成功但客户端仍把结果当异常，先检查 `success` 和 `isError` 是否对应。
- 如果更新了禁用工具但 `tools/list` 未变化，先重启服务，确认 `reinitialize()` 是否重新注入了 `disabled_tools`。
