# 服务与路由

本文档描述 v1.0 重构阶段已经确立的服务入口、会话分层与路由方向。

当前核心原则是：

- `Central Server` 是唯一对外公开的 MCP 入口。
- Godot 插件只承担编辑器侧代理角色。
- 插件内的 HTTP / attach 通道属于 Host 与编辑器之间的内部连接，不再是长期产品级公开入口。

---

## 服务入口

### 公开入口

公开 MCP 服务由 `central_server/` 提供：

- `CentralServerApplication`
- `CentralStdioMcpServer`
- `CentralToolCatalog`
- `CentralToolDispatcher`

外部客户端应连接：

```text
GodotDotnetMcp.CentralServer --stdio [--attach-host HOST] [--attach-port PORT]
```

### 编辑器侧入口

Godot 插件侧当前仍保留：

- `plugin/runtime/mcp_http_server.gd`
- `plugin/runtime/central_server_attach_service.gd`
- `plugin/runtime/server_runtime_controller.gd`

但它们的定位已经收口为：

- 编辑器内工具执行层
- attach / heartbeat / detach 会话端
- Host 私有转发通道
- 编辑器生命周期内部桥
  - Host 只会在 attach capabilities 明确声明 `editor_lifecycle` 时访问 `/api/editor/lifecycle`

---

## 工具分层

### Host-native

这类工具不依赖 Godot 正在运行：

- `workspace_*`
- `dotnet_*`

当前由 `CentralToolCatalog` 和 `CentralToolDispatcher` 统一公开和调度。

### Editor-attached

这类工具依赖实时编辑器状态：

- `system_*`

其中运行时自动化子集还额外包括：

- `system_runtime_control`
- `system_runtime_capture`
- `system_runtime_input`
- `system_runtime_step`

当前已经实现的主路径是：

1. 中心服务解析当前 MCP 会话对应的项目。
2. 如果已有可用 editor session，则直接转发。
3. 如果没有 session，则按项目拉起或复用 Godot。
4. 等待插件 attach。
5. attach 成功后转发 `system_*`。

对于 `workspace_project_open_editor` 和带 `autoLaunchEditor=true` 的 `system_*`，Host 现在还补了一条防重复打开约束：

1. 如果发现同一项目已经有未纳入当前 Host residency store 的 Godot editor 进程在运行，Host 会先等待它附着到当前 host。
2. 若该已有编辑器没有附着到当前 host，Host 会返回 `editor_already_running_external`。
3. 这时 Host 会明确拒绝再拉起第二个同项目编辑器实例，避免出现“双开编辑器但实际只有一条插件会话”的冲突状态。

当 `workspace_project_open_editor` 或 `system_*` 在这条会话协调链上失败时，Host 现在也会继续返回 `centralHostSession` 摘要，而不是只给裸错误码。这样失败载荷里仍能看到：

- 当前 `attachHost / attachPort / attachTimeoutMs`
- 当前 `activeProjectId / activeEditorSessionId`
- `centralHostSession.editorLifecycle.resolution`
  - 例如 `godot_executable_not_found`、`editor_attach_timeout`

对于编辑器生命周期控制（`workspace_project_close_editor / restart_editor`）：

1. 只有当 editor session 已 HTTP ready 且 capabilities 声明 `editor_lifecycle` 时，Host 才会尝试 graceful close / restart。
2. 若缺少该 capability，Host 会明确返回 `editor_lifecycle_unsupported`，而不是盲目探测内部接口。
3. force close 仍只允许用于 Host 自己拉起并持续跟踪的 resident editor。

对于 `system_runtime_*`，Host 还固定遵循两条额外规则：

1. 若编辑器已附着但项目尚未处于运行态，则返回 `runtime_not_running`，并明确建议先调用 `system_project_run`。
2. Host 不会为了 runtime 工具自动替用户启动游戏运行态；真正的“运行项目”仍由显式 `system_project_run` 触发。

当 `system_*` 通过 `Central Server` 成功转发时，Host 现在会在结果中补充 `centralHostSession` 摘要，至少明确：

- 当前 `projectId`
- 当前 `projectPath`
- 当前 `sessionId`
- 当前 `resolution`
- 当前 `activeProjectId`
- 当前 `activeEditorSessionId`

当前 Host 侧由以下对象共同完成这条链路：

- `CentralToolCatalog`
- `CentralToolDispatcher`
- `WorkspaceToolHandlerService`
- `EditorToolHandlerService`
- `WorkspaceEditorSessionToolHandlerService`
- `EditorAttachedToolForwardingService`
- `EditorSessionCoordinator`
- `EditorSessionAcquisitionService`
- `EditorLifecycleCoordinator`
- `EditorLifecycleStatusService`
- `EditorLifecycleRemoteStateService`
- `EditorLifecycleSummaryBuilder`
- `EditorLifecycleActionService`
- `EditorLifecycleGracefulActionExecutor`
- `EditorLifecycleForceActionExecutor`
- `EditorLifecycleActionResultFactory`
- `EditorSessionService`
- `EditorProcessService`
- `IExternalEditorProcessProbe`
- `EditorProcessResidencyService`
- `EditorProcessLaunchService`
- `EditorProcessTerminationService`

其中 `EditorLifecycleCoordinator` 当前已经退化为 façade：

- `EditorLifecycleStatusService` 负责 `workspace_project_status` 的项目解析、tracked process 同步与状态快照编排
- `EditorLifecycleRemoteStateService` 负责远端 editor lifecycle 状态探测与 unsupported/error 状态映射
- `EditorLifecycleSummaryBuilder` 负责 `editorLifecycle` 摘要与 resolution 归类
- `EditorLifecycleActionService` 负责 `workspace_project_close_editor / restart_editor` 的动作编排与前置校验
- `EditorLifecycleGracefulActionExecutor` 负责 graceful close / restart 的具体执行
- `EditorLifecycleForceActionExecutor` 负责 force close / restart 与必要时的 relaunch
- `EditorLifecycleActionResultFactory` 负责生命周期结果与错误 payload 的统一构造
- 对外工具名和主要 payload 形状保持不变，但 Host 内部不再把状态查询和生命周期动作堆在同一个巨型类里
- `CentralToolDispatcher` 当前也已经退化为 façade：
  - `WorkspaceToolHandlerService` 负责 `workspace_*` 中的 Host-native 项目与配置类工具
  - `EditorToolHandlerService` 负责 `workspace_project_status / close_editor / restart_editor`
  - `WorkspaceEditorSessionToolHandlerService` 负责 `workspace_project_open_editor`
  - `EditorAttachedToolForwardingService` 负责 `system_*` 转发
- `EditorSessionCoordinator` 当前同样已经退化为 façade：
  - `EditorSessionAcquisitionService` 负责项目解析、可执行文件决议、session 复用、attach 等待与必要时的 auto-launch
- `EditorProcessService` 当前也已把外部编辑器探测 seam 抽成 `IExternalEditorProcessProbe`，Windows 下默认由 `WindowsWmiExternalEditorProcessProbe` 提供实现，避免平台特化逻辑继续粘在主进程控制类里
- `EditorProcessService` 当前同样已经退化为 façade：
  - `EditorProcessResidencyService` 负责 tracked / untracked residency、状态构造与 store 迁移
  - `EditorProcessLaunchService` 负责真实 Godot 拉起与 launch residency 建账
  - `EditorProcessTerminationService` 负责 force stop 与等待退出

### 插件内部开发能力

以下能力目前主要用于插件内诊断和维护：

- `plugin_runtime_*`
- `plugin_developer_*`
- `plugin_evolution_*`

它们不应再默认作为 v1.0 的公开产品面继续扩张。

## 会话模型

中心服务需要显式区分三个概念：

- `registered project`
  已进入项目注册表的项目。
- `active project context`
  当前 MCP 会话正在操作的项目上下文。
- `attached editor session`
  已经成功附着到某个 Godot 编辑器实例的实时会话。

当前设计方向：

- `projectId` 负责项目主键和索引。
- `sessionId` 负责编辑器附着会话主键。
- `active project context` 与 `attached editor session` 可以独立存在。

当前实现状态：

- `EditorSessionService` 已使用 `sessionId` 作为内部主键。
- `projectId` 作为当前 attached session 的索引。
- `CentralWorkspaceState` 受控跟踪 `activeProjectId` 与 `activeEditorSessionId`，不再允许多服务直接写可变共享属性。

Godot 可执行路径策略：

- 中心服务只在常见目录中做轻量发现，不做广范围磁盘扫描。
- 如果没有已配置或可发现的 Godot 路径，Host 会返回 `godot_executable_not_found`，并附带结构化 `guidance`。
- 这份 `guidance` 的长期语义是：客户端或 Agent 应显式询问用户提供 Godot 编辑器路径，然后调用配置工具写入用户态配置。
- 机器相关路径不得写入发布产物、仓库默认配置或文档示例中的固定值。

---

## 路由责任

### Central Server

负责：

- 公开 `tools/list` 和 `tools/call`
- 项目选择与注册
- Godot 可执行路径解析
- 编辑器进程拉起
- attach 会话管理
- Host-native 工具调度
- Editor-attached 工具转发

### Godot 插件

负责：

- 在实时编辑器环境中执行工具
- 上报 attach / heartbeat / detach 状态
- 提供本地服务与诊断面板
- 维护插件侧工具运行时和 `user_*` 扩展

## 兼容与过渡

### 当前仍存在的过渡层

- 插件侧 `mcp_http_server.gd`

当前保留的过渡实现只剩插件侧 `mcp_http_server.gd`。
它仍然存在，是为了安全迁移和分批重构，不代表长期公开接口。

当前还新增了一条本地开发烟测主路径：

- `dotnet central_server/bin/Release/net8.0/GodotDotnetMcp.CentralServer.dll --smoke-system-session`
- `dotnet central_server/bin/Release/net8.0/GodotDotnetMcp.CentralServer.dll --smoke-system-session --auto-launch --project-root <path> [--godot-executable-path <path>]`
- `powershell -ExecutionPolicy Bypass -File scripts/smoke_central_server_system_session.ps1`
- `powershell -ExecutionPolicy Bypass -File scripts/smoke_central_server_system_session.ps1 -AutoLaunch -ProjectRoot <path> [-GodotExecutablePath <path>]`
- `powershell -ExecutionPolicy Bypass -File scripts/validate_central_server_release.ps1 -ProjectRoot <path> -GodotExecutablePath <path>`

它会在空闲端口环境下验证 Host 会话编排、attach 会话注册、`system_*` 转发与 `centralHostSession` 状态输出。
在 `--auto-launch` 成功路径下，烟测还会继续验证：

- `system_project_run`
- `system_runtime_control(action=enable)`
- `system_runtime_step(wait_frames=3, capture=true)`
- 截图文件落盘且为有效 PNG

其中 `--auto-launch` 会尝试走真实 Godot 拉起主路径；若当前机器没有配置或发现到 Godot 可执行文件，烟测会返回结构化 `skipped=true`，而不是给出模糊失败。
脚本默认还会把 `Central Server` 的存储目录重定向到仓库内的隔离 smoke 状态目录，避免污染真实用户配置；只有显式使用 `-UseUserProfileState` 时才会回到真实用户目录。

真实 auto-launch 路径还补了一条关键约束：

- Host 拉起 Godot 时，会通过环境变量把当前 attach listener 端点传给插件：
  - `GODOT_DOTNET_MCP_CENTRAL_SERVER_HOST`
  - `GODOT_DOTNET_MCP_CENTRAL_SERVER_PORT`
- 插件侧 `central_server_attach_service.gd` 会优先读取这两个运行时覆盖值，再回退到持久化设置。
- Host 还会为本次启动临时分配一个编辑器侧 MCP HTTP 端口，并通过环境变量传给插件：
  - `GODOT_DOTNET_MCP_SERVER_HOST`
  - `GODOT_DOTNET_MCP_SERVER_PORT`
- 这组端口只用于本次运行，不会回写到用户设置、仓库默认配置或发布产物中。

这些烟测的长期定位已经固定为：

- `reuse session smoke`
  - 长期作为 Host 会话编排与 `system_*` 转发的 CI 护栏
  - 必须在无真实 Godot 的环境里稳定跑通
- `auto-launch smoke`
  - 长期作为真实编辑器集成链路与发布验收护栏
  - 本地默认允许 `skipped=true`
  - 当需要严格验收时，可追加 `--require-auto-launch`，把 `skip` 也视为失败
- `validate_central_server_release.ps1`
  - 长期作为发布机标准化验收入口
  - 统一串联 `--health`、reuse smoke 和 `RequireAutoLaunch` 模式的 auto-launch smoke

### 长期方向

- 外部只保留 `Central Server`。
- 插件侧 transport 仅作为内部通道存在。
- 旧 `Dotnet Bridge` 产品壳已经移除，只保留 `host_shared/` 中必要的共享实现。

## 错误语义

围绕 editor-required 工具，中心服务需要返回结构化错误，而不是模糊失败：

- `project_not_selected`
- `project_not_registered`
- `godot_executable_not_found`
- `editor_launch_failed`
- `editor_attach_timeout`
- `editor_transport_unavailable`
- `editor_endpoint_unavailable`
- `editor_required`
- `runtime_not_running`
- `runtime_control_disabled`
- `runtime_session_lost`

目标是让客户端明确知道失败发生在：

- 项目上下文缺失
- Godot 路径解析失败
- 编辑器未启动
- 插件未附着
- 内部转发通道不可用
- 项目尚未进入运行态
- 运行时控制尚未显式启用
- debugger session 在命令过程中丢失

其中 `godot_executable_not_found` 还会额外提供：

- `guidance.askUserForGodotPath = true`
- 推荐向用户提问的 `suggestedUserPrompt`
- 推荐使用的配置工具：
  - `workspace_project_set_godot_path`
  - `workspace_godot_set_default_executable`
- 一条明确的发布约束：不要把机器相关 Godot 路径固化到发布产物中

---

## 当前实现边界

- 插件侧 HTTP 服务仍然存在，因此文档中会同时出现插件 transport 和中心服务 transport。
- 但对外说明必须始终把 `Central Server` 放在第一入口。
- 文档与代码会继续逐批收口，直到你明确确认“重构结束”为止。
