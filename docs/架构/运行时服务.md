# 运行时服务

本文档说明 `plugin/runtime/`、`plugin/config/` 和工具装载器如何协同工作，以及请求从 HTTP 入口进入后如何流向具体 executor。

---

## 服务分层

当前运行时大致分为四层：

```text
EditorPlugin 装配层
  ├─ plugin.gd
  ├─ plugin/plugin_bootstrap.gd
  ├─ plugin/plugin_dock_coordinator.gd
  ├─ plugin/plugin_runtime_coordinator.gd
  └─ plugin/plugin_action_router.gd

状态与配置层
  ├─ plugin/runtime/plugin_runtime_state.gd
  ├─ plugin/config/settings_store.gd
  └─ plugin/config/client_config_service.gd

展示与 Dock model 层
  ├─ plugin/presenters/dock_model_service.gd
  ├─ plugin/presenters/dock_presenter.gd
  └─ plugin/presenters/client_config_presenter.gd

服务与诊断层
  ├─ plugin/runtime/server_runtime_controller.gd
  ├─ plugin/runtime/mcp_http_server.gd
  ├─ plugin/runtime/mcp_tool_loader_supervisor.gd
  ├─ plugin/runtime/mcp_tool_rpc_router.gd
  ├─ plugin/runtime/mcp_editor_lifecycle_endpoint.gd
  ├─ plugin/runtime/mcp_tools_api_service.gd
  ├─ plugin/runtime/mcp_stdio_server.gd
  ├─ plugin/runtime/gdscript_lsp_diagnostics_service.gd
  ├─ plugin/runtime/plugin_reload_coordinator.gd
  ├─ plugin/runtime/mcp_runtime_bridge.gd（Autoload，主项目运行时事件通道）
  ├─ plugin/runtime/mcp_runtime_command_service.gd
  ├─ plugin/runtime/mcp_runtime_reply_service.gd
  ├─ plugin/runtime/mcp_runtime_fallback_store.gd
  ├─ plugin/runtime/mcp_editor_debugger_bridge.gd
  ├─ plugin/runtime/runtime_control_service.gd
  ├─ plugin/runtime/mcp_runtime_control_session_selector.gd
  ├─ plugin/runtime/mcp_runtime_control_error_mapper.gd
  ├─ plugin/runtime/mcp_runtime_control_reply_resolver.gd
  ├─ plugin/runtime/mcp_runtime_debug_store.gd
  └─ plugin/runtime/plugin_self_diagnostic_store.gd

工具装载与执行层
  ├─ tools/core/tool_loader.gd
  ├─ tools/tool_manifest.gd
  ├─ tools/tool_manifest_data.gd
  ├─ tools/<category>/executor.gd
  └─ tools/*_tools.gd（兼容 wrapper）
```

## 状态与配置层

### `plugin_runtime_state.gd`

这个对象当前提供轻量运行时状态容器，主要包括：

- 当前 settings 字典
- 当前 custom profiles 缓存
- 当前 CLI scope / Config platform / tab 等 UI 状态字段
- 仅承载 mutable runtime fields，本身不再负责 schema、profile 目录或持久化细节

它不再承担 permission、profile catalog 或 settings schema 的静态规范职责。

### `plugin_runtime_state_service.gd`

负责：

- 基于 `PluginSettingsSchema` 的 settings 初始化
- runtime state 的 load/save 编排
- custom profile 装载
- `current_cli_scope` / `current_config_platform` / `needs_initial_tool_profile_apply` 等状态字段归一化
- 将 state bootstrap / persistence 细节从 `plugin.gd` 中分离

### `plugin_bootstrap.gd`

负责：

- 创建插件侧 service / feature / presenter bundle
- 编排 `plugin_runtime_state_service.gd` 的 load/save 调用
- 配置 feature workflow 与 `dock_model_service.gd`
- 让 `plugin.gd` 更接近真正的 composition root，而不是继续承担 service factory

### `plugin_runtime_coordinator.gd`

负责：

- `user_tool_watch_service.gd` 的 stop/configure/start 编排
- 将用户工具 watcher 的外部刷新回调接到 `plugin_action_router.gd` 的显式 callback，而不是继续依赖 `plugin.gd` 的字符串方法名
- `central_server_process_service.gd` 的配置与 endpoint 检测刷新
- `central_server_attach_service.gd` 的配置与启动
- 本地 `Central Server` endpoint 可达性与 attach 触发协调

### `plugin_dock_coordinator.gd`

负责：

- Dock 场景创建、挂载与销毁
- stale dock 清理与重复实例检测
- `FileDialog` 节点创建 / 清理与回调接线
- Dock signal wiring 的统一编排

它让 `plugin.gd` 不再继续直接持有整段 dock 生命周期细节。

### `plugin_action_router.gd`

负责：

- 将 Dock / Config / Server / User Tool 等 UI 事件统一映射到对应 feature
- 作为 `plugin.gd` 的动作转发层，集中承接 show message / confirmation、dock refresh 与各类 action handler
- 为 `user_tool_watch_service.gd` 提供显式 callback 入口，避免 watcher 继续依赖 `plugin.gd` 的字符串方法名
- 将用户工具 catalog refresh 与 runtime reload 相关的辅助入口收口到单一协作者

### `plugin_tool_bridge_service.gd`

负责：

- 统一 `plugin_runtime` / `plugin_developer` / `plugin_evolution` 的工具桥接
- runtime / developer / evolution 工具入口分发
- usage guide 生成
- 将 `plugin.gd` 中原本分散的 `*_from_tools` / `runtime_*` facade 收口为稳定桥接层

### `tool_permission_policy.gd`

负责：

- permission level 常量
- plugin category 权限映射
- tool/domain permission 计算
- tool name 到 category 的解析
- permission 配置一致性校验

### `tool_profile_catalog.gd`

负责：

- builtin tool profiles
- custom profile storage dir

### `plugin_settings_schema.gd`

负责：

- settings path
- default settings
- collapsed domain / system tool 默认值
- settings 初始化所需的 category / domain 默认输入

### `settings_store.gd`

负责：

- 读取 / 保存插件设置
- 读取 / 保存自定义 profile
- 重命名 / 删除 profile
- 导入 / 导出工具配置 JSON

当前持久化边界：

- `user://godot_dotnet_mcp_settings.json`
- `user://godot_dotnet_mcp_tool_profiles/*.json`
- 用户指定的导入 / 导出 JSON 路径

### `client_config_service.gd`

负责：

- 作为 Config 页配置链门面
- 统一委托 serializer / inspection / transaction / launcher adapter
- 保持 UI feature 与具体文件写盘 / CLI 行为分离

### `client_install_detection_service.gd`

负责：

- 客户端安装路径探测
- 客户端运行状态探测
- 手工 executable path 与自动探测结果归一化
- 短期检测缓存与失效控制

当前由 `config_feature.gd`、`ui_state_feature.gd` 和 `dock_model_service.gd` 直接使用，不再经 `plugin.gd` wrapper 中转。

### `dock_model_service.gd`

负责：

- 聚合 Dock 所需的运行时 context
- 对 `tools_by_category` 做 permission filter
- 汇总 self-diagnostic、`Central Server`、user tool watch 与 config tab 安装检测状态
- 把 context 统一交给 `dock_presenter.gd`

### `dock_presenter.gd`

负责：

- 把 Dock context 映射为最终 UI model
- 解析 profile 描述、中心服务反馈文本与 Config 页展示辅助
- 将 `client_config_presenter.gd` 暴露为 Config 页客户端视图子协作者

## 服务与诊断层

### `server_runtime_controller.gd`

这是 `plugin.gd` 与 `MCPHttpServer` 之间的代理层，负责：

- 创建 / 挂载 server 节点
- 在 `soft_reload_plugin` 时重建 controller 与 server 节点
- 启动 / 停止 / 重启 server
- 将设置与禁用工具列表同步给 server
- 向入口抛出 `server_started`、`server_stopped`、`request_received`

它的存在让 `plugin.gd` 不需要直接操作 TCP server 细节。

### `mcp_http_server.gd`

这是实际的内嵌 HTTP + MCP 服务节点，负责：

- 监听 TCP 端口
- 接收 HTTP 请求
- 解析 JSON-RPC / MCP 请求体
- 将 HTTP path 分发委托给 `mcp_http_request_router.gd`
- 将 JSON-RPC method 分发委托给 `mcp_json_rpc_router.gd`
- 将 `/health`、JSON-RPC 响应、HTTP 响应写出与 JSON 清洗委托给 `mcp_http_response_service.gd`
- 将 editor lifecycle close / restart 的确认、accepted payload 与 deferred 执行委托给 `mcp_editor_lifecycle_action_service.gd`
- 将 editor lifecycle 状态快照构建委托给 `mcp_editor_lifecycle_state_builder.gd`
- 通过 `mcp_tool_rpc_router.gd` 委托 `tools/list`、`tools/call`、tool 名称解析与结果归一化
- 维护连接数、请求数、最近请求方法和时间
- 通过 `mcp_tool_loader_supervisor.gd` 委托 tool loader 生命周期、自恢复和状态摘要
- 通过 `mcp_editor_lifecycle_endpoint.gd` 委托 editor lifecycle 请求解析与动作分派
- 通过 `mcp_tools_api_service.gd` 委托 `/api/tools` 快照构建
- 通过 `plugin.gd.get_tool_access_provider()` 将 permission provider 直接桥接给 `tool_loader.gd`
- 持有 `runtime_control_service.gd`，为 `system_runtime_*` 提供编辑器侧 runtime command 协调入口

### `mcp_tool_loader_supervisor.gd`

负责：

- 持有当前 `MCPToolLoader`
- tool loader 初始化 / 重建 / 恢复
- disabled tools 同步
- loader health 分类
- loader summary / status 视图生成
- 将 `mcp_http_server.gd` 从“协议处理 + loader 生命周期”混合体继续收口

### `mcp_tool_rpc_router.gd`

负责：

- `tools/list`
- `tools/call`
- tool 名称解析
- tool result normalization
- 将 `mcp_http_server.gd` 从“顶层方法分发 + 工具调用语义”继续收口为“顶层方法分发”

### `mcp_editor_lifecycle_endpoint.gd`

负责：

- 解析 `/api/editor/lifecycle` POST body
- 校验 `action` 字段
- 分派 `status / close / restart`
- 为 editor lifecycle 桥提供更稳定的结构边界

它当前主要服务于两类场景：

- `mcp_http_server.gd` 内部的 lifecycle 请求处理
- 插件 headless contract tests 对 lifecycle 结构契约的稳定验证

### `mcp_tools_api_service.gd`

负责：

- 构建 `/api/tools` 快照
- 汇总 `tools`
- 汇总 `domain_states`
- 汇总 `tool_loader_status`
- 汇总性能摘要

它的作用是把 `mcp_http_server.gd` 中“HTTP 顶层路由”与“工具快照拼装”继续拆开。

### `mcp_http_request_router.gd`

负责：

- `POST /mcp`
- `GET /health`
- `GET /api/tools`
- `GET /api/editor/lifecycle`
- `POST /api/editor/lifecycle`
- `OPTIONS` CORS

它的作用是把 `mcp_http_server.gd` 中“TCP/HTTP 传输层”与“path 级请求分发”继续拆开。

### `mcp_json_rpc_router.gd`

负责：

- `initialize`
- `tools/list`
- `tools/call`
- `ping`
- notification 无响应语义
- method-not-found 错误建模

它的作用是把 `mcp_http_server.gd` 中“MCP 请求体解析”与“JSON-RPC method 级分发”继续拆开。

### `mcp_http_response_service.gd`

负责：

- JSON-RPC success / error 响应构造
- `/health` 快照拼装
- CORS 预检响应
- HTTP 响应写出
- JSON 可序列化清洗

它的作用是把 `mcp_http_server.gd` 中“传输层”与“响应格式化/健康投影”继续拆开。

### `mcp_editor_lifecycle_action_service.gd`

负责：

- editor lifecycle `close / restart` 的确认语义
- accepted payload 构造
- deferred shutdown / finalize 执行
- attach service 停止与保存场景前置动作

它的作用是把 `mcp_http_server.gd` 中“lifecycle 请求接受”与“lifecycle 动作执行链”继续拆开。

### `mcp_editor_lifecycle_state_builder.gd`

负责：

- editor lifecycle 状态快照构建
- `openScenes / dirtyScenes / currentScenePath` 投影
- lifecycle hint 拼装

它的作用是把 `mcp_http_server.gd` 中“lifecycle 路由/动作接受”与“编辑器状态快照采集”继续拆开。

### `mcp_stdio_server.gd`

这是与 HTTP server 共享同一套 `MCPToolLoader` 的 stdio transport，负责：

- 通过 `Content-Length` 帧读取 stdin / 写出 stdout
- 复用同一份工具定义、禁用工具与可见工具集合
- 在 `transport_mode = stdio / both` 时与 HTTP transport 并行工作

### `gdscript_lsp_diagnostics_service.gd`

这是 GDScript LSP 诊断的后台调度层，负责：

- 统一持有 `lsp_client.gd`
- 管理活动请求、缓存与最近完成状态
- 为 `system_script_analyze` 提供后台诊断结果
- 为 `plugin_runtime_state(action=get_lsp_diagnostics_status)` 提供详细自检快照

### `plugin_reload_coordinator.gd`

负责完整插件重载的跨实例协调，包括：

- 在旧实例退出前存储待恢复状态
- 在新实例进入后恢复页签与焦点
- 降低直接 `soft_reload` 造成的 UI 悬挂状态

### `mcp_runtime_bridge.gd`

以 Autoload 形式安装到主项目（注入 `project.godot`），在主项目运行期间作为事件通道使用。

当前这条链已经开始拆成更小协作者：

- `mcp_runtime_command_service.gd`：负责 `status / capture / input / step` 的动作分发与执行。
- `mcp_runtime_reply_service.gd`：负责 success / error reply、`runtime_context`、`runtime_state` 与 hint 补全。
- `mcp_runtime_fallback_store.gd`：负责 fallback 事件的内存缓存、裁剪、持久化与读取。

**事件 / 命令 API**：

- `emit_event(event_type, data)` — 发送结构化事件
- `emit_log(message, level)` — 发送日志消息
- `emit_info(message)` / `emit_warning(message)` / `emit_error(message)` — 快捷等级入口
- `register_message_capture("godot_mcp/runtime_command", ...)` — 捕获编辑器侧 runtime command
- `send_message("godot_mcp/runtime_reply", payload)` — 回传命令结果

**双向协议设计**：

- `godot_mcp/runtime_command` — 编辑器到运行时的控制命令，统一带 `request_id`
  - `action=capture`：统一截图命令；`frame_count=1` 为单帧，`frame_count > 1` 时按 `interval_frames` 做低频序列
- `godot_mcp/runtime_reply` — 运行时到编辑器的命令响应，统一带 `request_id`、`ok`、`error?`、`data?`
  - 失败时统一补充 `data.runtime_context`、可用时补充 `data.runtime_state`，并提供 `data.hint`
- `godot_mcp/runtime_event` — 结构化事件（含 event_type、data、timestamp）
- `godot_mcp/runtime_log` — 日志消息（含 level、message、timestamp）

**Fallback 机制**：当 EngineDebugger 不活跃时（主项目未在编辑器调试模式下运行），事件写入 `user://godot_mcp_runtime_bridge_events.json`，并由独立 fallback store 负责缓存、裁剪与持久化。

**缓冲上限**：最多缓存 300 条事件，超出时丢弃最早记录。

**运行时自动化能力**：

- 单帧截图与低频帧序列，输出到 `user://godot_mcp_runtime_captures/<session_id>/`
- `InputMap action` 与原始 `InputEventKey` 注入
- 标准化 `step` 闭环：输入 -> 等待若干帧 -> 截图 -> 返回轻量 `runtime_state`

### `mcp_editor_debugger_bridge.gd`

编辑器侧接收器，注册到 `EditorDebuggerPlugin`，监听 EngineDebugger 消息，将主项目发来的 `godot_mcp/runtime_event` 和 `godot_mcp/runtime_log` 消息转发至 `MCPDebugBuffer` 和 `MCPRuntimeDebugStore`，使插件侧工具（`debug_runtime_bridge`、`system_runtime_diagnose`）可以读取主项目运行时事件。

它同时暴露当前 active debugger session 给 `runtime_control_service.gd`，供运行时控制链路下发 `send_message()`。

### `runtime_control_service.gd`

运行时控制协调层，负责：

- 持有当前可用的 `EditorDebuggerSession`
- 管理 `armed / disarmed` 安全闸状态
- 生成 `request_id` 并发送 `godot_mcp/runtime_command`
- 等待对应 `runtime_reply`
- 处理超时、会话丢失、插件重载与 server 停止时的自动失效

它的长期职责是“编辑器侧编排”，而不是截图或输入本身的实际执行者；真正的截图、输入注入和 step 仍在运行中的项目里完成。

当前错误模型中，编辑器侧协调层会为失败结果统一补充 `data.editor_context` 和 `data.hint`；当失败来自运行时 reply 时，外层可以同时拿到 `editor_context` 与 `runtime_context`。

当前这条链已经开始拆成更小协作者：

- `mcp_runtime_control_session_selector.gd`：负责首选 session 选择、commandable 判断、session snapshot 与等待可用 session。
- `mcp_runtime_control_error_mapper.gd`：负责 `editor_context`、hint 与统一错误 payload。
- `mcp_runtime_control_request_coordinator.gd`：负责 request_id、pending request 生命周期、runtime reply 回填、fallback 等待与 session-lost 失效标记。
- `mcp_runtime_control_reply_resolver.gd`：负责 fallback reply 搜索与 runtime reply 归一化。

### `mcp_runtime_debug_store.gd`

负责保留运行时桥的近期事件、会话状态和错误摘要，供 `debug_runtime_bridge` 与 Dock 自检使用。

### `plugin_self_diagnostic_store.gd`

负责插件自身 incident / operation 的结构化记录，用于：

- Dock 自检卡片
- `plugin_runtime_state(action=get_lsp_diagnostics_status)`：详细运行态自检快照
- `project_state(include_runtime_health=true)`：轻量健康摘要

## 工具装载层

### `tool_manifest_data.gd`

负责承载工具目录的纯静态数据，包括 builtin tool entries、category、domain、默认折叠域与 exposed category。

### `tool_manifest.gd`

负责 `MCPToolManifest` 的访问与校验入口。它通过 preload `tool_manifest_data.gd` 提供稳定的读取 API，并继续作为 `tool_loader.gd` 的直接运行时入口。

### 权限 provider 链路

当前工具权限链路已经收口为：

```text
plugin.gd
  └─ get_tool_access_provider()
       └─ plugin/features/tool_access_feature.gd
            └─ tool_loader.gd
```

这意味着：

- `tool_loader.gd` 不再经由 `plugin.gd` 的 permission facade 查询可见性或拒绝提示
- `tool_access_feature.gd` 成为唯一的插件侧 permission provider
- `plugin.gd` 只保留稳定桥接入口，而不再承担权限中转实现

### `tool_manifest.gd`

负责工具目录的访问层，当前对外稳定提供：

- builtin tool entries
- all tool categories（仅保留当前主分类）
- domain defs
- default collapsed domains
- exposed categories

### `tool_loader.gd`

负责：

- 读取 registry
- 维护 `category -> entry / runtime / tool definitions`
- 初始化时预加载启用 category
- 热重载单个 domain 或全部 domain
- 汇总 load error、reload status、tool call metrics
- 执行 permission gate 与 disabled tools gate
- 持有并重建 GDScript LSP diagnostics service
- 协调 `tool_exposure_service.gd` 生成 Dock、`/api/tools` 与 MCP `tools/list` 共用的当前可见工具集合

### `tool_exposure_service.gd`

负责：

- 装饰 tool definition
- 生成 tools by category 视图
- 过滤 exposed tool 集合
- 汇总 domain state 与 tool loader status
- 将工具系统的“目录视图层”从 `tool_loader.gd` 中分离

### `tool_metrics_service.gd`

负责：

- startup / definition scan / preload 指标记录
- reload metrics 聚合
- per-tool usage 指标聚合
- performance summary 与 usage stats 视图生成
- 将工具系统的“统计视图层”从 `tool_loader.gd` 中分离

### `tool_diagnostic_service.gd`

负责：

- load error list 存取
- registry refresh 阶段错误收集
- duplicate category 诊断收集
- `record_load_error`
- `sync_load_error_incidents`
- `record_reload_incident`
- 将工具系统的“诊断事件层”从 `tool_loader.gd` 中分离

### `tool_runtime_service.gd`

负责：

- `ensure_tool_definitions`
- `ensure_runtime_loaded`
- `instantiate_executor`
- `extract_tool_definitions`
- `unload_runtime`
- 将 executor runtime 的底层生命周期从 `tool_loader.gd` 中分离

### `tool_reload_service.gd`

负责：

- `reload_domain`
- `reload_all_domains`
- reload 前 runtime / definitions 快照与失败回滚
- reload 成功后的 runtime context / LSP diagnostics 刷新
- 将 reload orchestration 从 `tool_loader.gd` 中分离

### `tools/<category>/executor.gd`

每个 executor 对外暴露统一的 `execute(tool_name, args)` 入口，域内再按：

- `tool_name`
- `args.action`

做二级分发。

### `tools/*_tools.gd`

这些文件主要用于兼容层或共享 helper，例如：

- `base_tools.gd`
- `scene_tools.gd`
- `node_tools.gd`（当前为兼容薄 wrapper，稳定入口已切到 `tools/node/executor.gd`）
- `mcp_script_parser.gd`
- `mcp_json_utils.gd`

新结构优先使用 `tools/<category>/executor.gd`，根目录 `*_tools.gd` 当前主要保留为共享逻辑或过渡壳，不再作为新增能力的首选落点。

## 一次工具请求的完整链路

```text
Client
  -> HTTP / MCP Request
  -> MCPHttpServer
  -> MCPToolLoader.execute_tool(category, tool_name, args)
  -> category executor.gd
  -> domain helper / Godot API
  -> Dictionary result
  -> MCPHttpServer 封装响应
  -> Client
```

执行过程中会附带以下约束：

1. 先做 permission level 校验。
2. 再判断目标 category 是否已装载；未装载则按需加载。
3. 再执行工具本体。
4. 执行成功与失败都会计入调用统计。
5. 若装载或执行失败，会在结果里补充结构化错误字段。
