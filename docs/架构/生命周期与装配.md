# 生命周期与装配

本文档描述插件入口 `plugin.gd` 如何装配运行时对象、注册 Dock、接入 Autoload 与调试桥，并说明完整重载与退出时的清理顺序。

---

## 核心入口

当前插件入口是 `addons/godot_dotnet_mcp/plugin.gd`，类型为 `EditorPlugin`。它自身不再承载所有业务逻辑，而是作为薄装配层，把以下对象组合起来：

- `LocalizationService`
- `PluginRuntimeState`
- `SettingsStore`
- `ServerRuntimeController`
- `ToolCatalogService`
- `ClientConfigService`
- `UserToolService`
- `PluginReloadCoordinator`
- `MCPEditorDebuggerBridge`
- `MCPRuntimeDebugStore`
- `plugin/features/server_feature.gd`
- `plugin/features/config_feature.gd`
- `plugin/features/ui_state_feature.gd`
- `plugin/features/user_tool_feature.gd`
- `plugin/features/reload_feature.gd`
- `plugin/features/tool_profile_feature.gd`
- `plugin/features/tool_access_feature.gd`
- `plugin/features/self_diagnostic_feature.gd`
- `PluginSelfDiagnosticStore`
- `plugin/presenters/dock_model_service.gd`
- `plugin/presenters/dock_presenter.gd`
- `plugin/presenters/client_config_presenter.gd`
- `ui/mcp_dock.tscn`

这意味着入口主要负责“装配、同步、桥接”，不直接承担复杂 UI 逻辑或工具实现。

---

## `_enter_tree()` 装配顺序

`plugin.gd._enter_tree()` 当前执行顺序如下：

1. 清空上一轮插件自诊断缓存。
2. 重新创建 `PluginRuntimeState`，避免热重载后复用退出阶段已清空的旧状态容器。
3. 调 `_refresh_service_instances()`，保证本轮入口持有新的服务对象实例。
4. 调 `_load_state()` 读取设置文件与自定义 profile。
5. 校验 permission 配置与 domain/category 一致性。
6. 重置并初始化本地化服务，应用当前语言。
7. 将日志等级同步给 `MCPDebugBuffer`。
8. 把当前 `settings` 交给 `ServerRuntimeController.attach()`。
9. 安装运行时桥接 Autoload：`MCPRuntimeBridge`。
10. 安装编辑器调试桥：`MCPEditorDebuggerBridge`。
11. 实例化并挂载 `ui/mcp_dock.tscn`。
12. 如果是首次载入设置，按默认 profile 推导 `disabled_tools`。
13. 首轮 `_refresh_dock()` 投影 UI model。
14. 开启 `_process()` 轮询。
15. 如果 `auto_start=true`，自动启动内嵌 MCP 服务。
16. 如有重载前保存的焦点快照，则恢复 Dock 页签与焦点。

入口当前把 `PluginActionRouter`、`PluginBootstrap`、`PluginDockCoordinator`、`PluginRuntimeCoordinator` 作为稳定成员协作者持有，不再在运行路径中做惰性初始化分支。

---

## `_exit_tree()` 清理顺序

退出时不直接依赖 Godot 的“节点自然释放”，而是显式按顺序清理：

1. 关闭 `_process()`。
2. 保存当前设置。
3. 移除 Dock。
4. 卸载编辑器调试桥。
5. 让 `ServerRuntimeController.detach()` 停止服务并释放 server 引用。
6. 重置本地化服务单例。
7. 清空入口持有的服务引用，避免悬挂实例被下一次热重载复用。

这个顺序的目标是降低完整重载和插件禁用过程中的残留状态。

## Dock 装配链路

Dock 装配由 `_create_dock()` 执行，内部步骤为：

1. `_remove_dock()` 清理旧实例。
2. `_remove_stale_docks()` 处理热重载后可能残留的旧 Dock。
3. 加载 `ui/mcp_dock.tscn`。
4. 实例化根节点后，通过 `_wire_dock_signals()` 连接所有页签信号。
5. 调 `add_control_to_dock(DOCK_SLOT_RIGHT_UL, _dock)` 注册到编辑器。
6. 如发现多实例，则记录自诊断 incident。

入口和 Dock 的关系是单向的：

- Dock 不持有服务对象
- Dock 只暴露信号和 `apply_model()`
- 信号在装配阶段由 `plugin.gd` 绑定到 feature 或稳定桥接方法，再流向具体服务

## 运行时桥与调试桥

### 运行时桥 Autoload

`_ensure_runtime_bridge_autoload()` 会保证项目中存在：

- 名称：`MCPRuntimeBridge`
- 路径：`res://addons/godot_dotnet_mcp/plugin/runtime/mcp_runtime_bridge.gd`

如果同名 Autoload 已被其它脚本占用，插件不会强制覆盖，而是：

- 记录运行时桥状态为失败
- 在自诊断中写入冲突 incident
- 保持插件其余部分继续工作

### 编辑器调试桥

`_install_editor_debugger_bridge()` 会实例化 `MCPEditorDebuggerBridge` 并注册为 `EditorDebuggerPlugin`，用于把运行中的主项目调试会话摘要同步到插件可读取的数据结构中。

---

## 状态轮询与 model 刷新

插件当前没有把每个服务状态变化都做成深层依赖通知，而是采用轻量轮询：

- `plugin.gd._process()` 每 `0.5s` 调一次 `_refresh_dock()`
- `_refresh_dock()` 重新构建统一 model，并把 Dock context 聚合委托给 `dock_model_service.gd`，再交给 presenter 脚本做展示映射
- `mcp_dock.gd.apply_model()` 再把 model 分发到三个页签

这种做法的优点是：

- 入口逻辑简单，状态同步路径清晰
- 语言、缩放、服务状态与统计都能统一刷新
- UI 控制器保持近似纯显示层

## 完整重载策略

插件支持两类重载：

### 1. 运行时重载

通过 `plugin_runtime_reload` 或 server 页触发，刷新工具域或 server，而不销毁整个插件实例。

### 2. 完整插件重载

通过 `PluginReloadCoordinator` 驱动 Godot 插件层的：

1. `set_plugin_enabled(false)`
2. `set_plugin_enabled(true)`

完整重载前会记录：

- 当前宿主 Dock 页签
- `MCPDock` 内部当前 Tab
- 当前焦点控件的相对路径

重载后会重新激活宿主 Dock，再恢复内部页签与焦点。

## 边界与职责

### `plugin.gd` 负责

- 生命周期
- 服务对象装配
- Dock 场景装配
- UI signal -> feature / 服务调用桥接
- feature workflow 配置
- Dock model 协作者装配与 presenter 调用
- 对 `tool_loader` 暴露稳定的 permission provider 入口
- 设置读写触发点
- 自诊断 incident 记录

### `plugin.gd` 不负责

- 具体 UI 布局
- 单个页签的控件状态机
- `Central Server` / Config / User Tool / Runtime Reload / Profile / Config Exchange / Tool Access / Self Diagnostic 交互步骤编排
- Dock 页签状态写回、语言切换、Tree collapse 持久化与复制提示编排
- Dock context 聚合、permission filter 与安装检测细节
- `tool_loader` 的权限查询实现
- 工具协议分发
- profile 文件读写细节
- 客户端配置合并逻辑
- 工具域执行逻辑
